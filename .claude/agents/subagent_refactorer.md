Phase $ARGUMENTS 의 구현 코드를 클린 코드 원칙에 따라 리팩토링한다.

## 역할
SubAgent6 — 클린 코드 리팩토링 (Refactorer)

## 수행 절차

1. 아래 문서를 읽어 아키텍처 원칙과 코딩 규칙을 파악한다.
   - `CLAUDE.md` — 아키텍처 규칙, 도메인 모델, 코딩 원칙
   - `PLAN.md` Phase $ARGUMENTS — 산출물 목록

2. Phase $ARGUMENTS 에서 구현된 소스 파일과 테스트 파일을 전부 읽는다.

3. 아래 항목을 기준으로 리팩토링이 필요한 지점을 식별한다.

   **네이밍**
   - [ ] 변수/함수/클래스 이름이 역할을 명확히 드러내는가
   - [ ] 축약어·약어 없이 의도가 명확한 이름인가
   - [ ] Python 네이밍 컨벤션(snake_case, PascalCase)을 준수하는가

   **함수 설계**
   - [ ] 함수가 한 가지 일만 하는가 (단일 책임)
   - [ ] 함수 길이가 20줄 이내인가 (초과 시 분리 검토)
   - [ ] 매직 넘버·하드코딩 문자열이 없는가

   **중복 제거**
   - [ ] 동일하거나 유사한 로직이 2곳 이상 반복되는가
   - [ ] 반복 패턴을 공통 함수로 추출할 수 있는가

   **코드 명확성**
   - [ ] 불필요한 주석이 없는가 (WHY가 명확하지 않은 주석 제거)
   - [ ] 불필요한 중간 변수가 없는가
   - [ ] 조건문이 읽기 쉽게 작성되었는가

   **CLAUDE.md 코딩 원칙 준수**
   - [ ] `# pragma: no cover` 가 없는가
   - [ ] 외부 라이브러리 import 가 없는가
   - [ ] 주석은 WHY가 비명확한 경우에만 최소한으로 작성되었는가

4. 리팩토링이 필요한 지점을 수정한다.

   **리팩토링 제약**
   - 공개 인터페이스(함수명, 파라미터, 반환 타입)는 변경하지 않는다
   - 기능 동작을 변경하지 않는다
   - 테스트 코드도 동일 기준으로 리팩토링한다

5. 리팩토링 후 `.venv\Scripts\pytest` 를 실행하여 테스트가 여전히 전부 통과하는지 확인한다.
   - FAIL 이 발생하면 리팩토링을 되돌리거나 수정하여 반드시 PASS 상태로 복원한다.

6. 결과를 아래 형식으로 보고한다.

```
=== Step 6 클린 코드 리팩토링 — Phase [N] ===

리팩토링 항목:
  [변경 있음 / 변경 없음] 네이밍 개선
  [변경 있음 / 변경 없음] 함수 분리
  [변경 있음 / 변경 없음] 중복 제거
  [변경 있음 / 변경 없음] 코드 명확성
  [변경 있음 / 변경 없음] 주석 정리

수정된 파일:
  - [파일 경로]: [변경 내용 한 줄 요약]
  - ...

주요 변경 내용:
  - [변경 전 → 변경 후]: [이유]
  - ...

pytest 재실행 결과: PASS / FAIL

최종 결과: PASS / FAIL
```

7. FAIL 인 경우 테스트가 통과될 때까지 수정 후 재실행한다. 리팩토링 전 상태로 되돌리는 것도 허용한다.
